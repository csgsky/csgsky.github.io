Android 中的 SurfaceView 详解

SufaceView 为什么会出现

Android 中提供了 View 进行绘制处理，而且 View 满足了大部分使用场景，View 通过刷新来重新绘制视图，系统发出VSYNC 信号来进行屏幕绘制，刷新的时间间隔是 16ms，如果我们 View 的刷新工作能够在 16ms 里面完成，且刷新不是那么频繁，那么使用 View 是没有任何问题滴，但是如果我们绘制一些比较复杂的过程，而且更新还十分频繁，这个时候使用 View 就会造成界面卡顿，影响用户体验，因此才会出现 SufaceView。关于 VSYNC 的介绍，以及为什么会出现卡顿，请参考： https://www.cnblogs.com/xiaoniuzai/p/6419150.html 这边做一个简单的介绍。

在计算系统中，CPU 将计算好的内容给 GPU，GPU 渲染完成后将渲染结果放在缓冲区，随后视频控制器会按照 VSYNC 信号来逐行读取缓冲区中的数据，经过可能的数模转换传递给显示器。

一般情况下，缓冲区只有一个，这个时候读取和刷新都会有比较大的效率问题，为了解决这个问题，显示系统通常会引入两个缓冲区，即双缓冲区 GPU 将渲染好的数据交替放置在两个缓冲区，让视频控制器读取，大大的提高了效率。

双缓冲虽然能解决效率问题，但会引入一个新的问题。当视频控制器还未读取完成时，即屏幕内容刚显示一半时，GPU 将新的一帧内容提交到帧缓冲区并把两个缓冲区进行交换后，视频控制器就会把新的一帧数据的下半段显示到屏幕上，造成画面撕裂现象。为了解决这个问题，GPU 通常有一个机制叫做垂直同步（简写也是 V-Sync），当开启垂直同步后，GPU 会等待显示器的 VSync 信号发出后，才进行新的一帧渲染和缓冲区更新。这样能解决画面撕裂现象，也增加了画面流畅度，但需要消费更多的计算资源，也会带来部分延迟。

SufaceView 和 View 对比

  • View 比较适用于主动更新，比如一些自动执行的动画等等。SurfaceView 更适用于一些被动更新，比如频繁刷新界面
  • View 在主线程对 UI 进行更新，SurfaceView 开启一个子线程对页面进行刷新
  • View 在绘制的时候没有实现双缓冲，SurfaceView 在底层机制中实现双缓冲技术
    双缓冲技术是游戏开发中的一个重要的技术。当一个动画争先显示时，程序又在改变它，前面还没有显示完，程序又请求重新绘制，这样屏幕就会不停地闪烁。而双缓冲技术是把要处理的图片在内存中处理好之后，再将其显示在屏幕上。双缓冲主要是为了解决 反复局部刷屏带来的闪烁。把要画的东西先画到一个内存区域里，然后整体的一次性画出来。
